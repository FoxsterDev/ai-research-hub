Architecture without hidden layers

AIRoot stays intentionally routerless. That is a feature, not a limitation. It keeps reusable public logic separate from host-specific routing and project truth.

The public layer gives you shared protocols such as `xuunity`, reusable setup docs, templates, and design artifacts. A real host repo then adds runtime routing, private overlays, mutable operations state, and project-local memory.

Keep public-safe logic in AIRoot

Shared execution modules, reusable setup, generic operations docs, and architecture visuals belong here.

Keep host-private routing outside AIRoot

Repo routers, private protocol families, mutable operations state, and internal overlays should stay in the host repository.

Keep project truth local to projects

Project routers, project memory, and generated outputs remain the durable source of truth for real implementation work.

Three practical adoption paths

Different visitors need different entrypoints. AIRoot should explain itself to all three without mixing public docs with private runtime assumptions.

01

I am evaluating the repo

Read the handbook, read the visual map, and decide whether the public module boundaries match the engineering workflows you want to support.

02

I want to mount it into a host

Use the integration guide and setup index. AIRoot gives you the reusable core. Your host repo still owns runtime routing, private overlays, and mutable state.

03

I want to extend the protocol system

Add public-safe reusable behavior to AIRoot. Add host-private families and internal overlays only in the host repository that consumes it.
